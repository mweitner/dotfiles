# AI / Automation Guidelines for dotfiles

Purpose: Provide concise rules for AI agents and automated tooling that modify files in this repository.

Scope:

- Edits to shell scripts in `shell/`, service snippets in `systemd/`,
  config files under `sway/`, `i3/`, `fish/`, and documentation files.

Rules:

- Keep changes minimal and focused. Prefer one logical change per commit.
- Do not invent credentials, secrets, or private hostnames. If a secret is
 required, add a clear placeholder and instruct the maintainer to populate it
 from a secure store.
- Follow repository style: Markdown line length <= 120 characters.
- Validate shell changes with `shellcheck` and `shfmt` where possible before proposing changes.
- For any change that modifies installer scripts (`install-*.sh`) or
 system-wide configuration, include a short manual verification checklist in
 the commit message or PR description.
- All AI-generated edits must be human-reviewed before merge.
- Add tests or lint fixes where practical (for example, add `shfmt`
 formatting changes in the same commit that introduces other shell edits).

Operations that require explicit human approval (do not do automatically):

- Adding new `sudo` commands, new systemd service files, or changes that write to `/etc` paths.
- Adding external package repositories or importing GPG keys.

Contact: Refer to repository `CODEOWNERS` or repo maintainers for approvals and sensitive changes.

## Markdown linting rules (for AI agents)

The repository uses `.markdownlint.yaml` at the repo root as the authoritative
source for markdown lint rules. AI agents should follow these conventions when
generating or editing `.md` files in this repo.

- `MD013` (line-length): enforced at 120 characters for regular text.
 Code blocks, links/URLs, and tables are exempt.
- `MD033` (no-inline-html): enabled.
- `MD025` (single-title): enabled (single top-level heading per file).
- `MD040` (fenced-code-language): required for AI-authored or AI-edited markdown.
 Every fenced block must include a language label (for example `bash`, `yaml`,
 `json`, `python`, `markdown`, `text`).
- `MD041` (first-line-heading): enabled.
- `MD045` (no-alt-text): enabled.

Agent Checklist before committing markdown changes:

1. Run `pre-commit run markdownlint --all-files` locally to surface issues.
2. If `markdownlint` reports problems that cannot be fixed automatically,
   present the exact errors and suggested edits to the human reviewer.
3. Add `[AI-ASSISTED]` to commit messages when changes were generated or modified by AI.

### Fenced code block rule (MD040)

Rule: all fenced code blocks must include a language identifier.

When adding or editing a fenced block:

- Choose the language from content: `bash`, `yaml`, `json`, `python`,
 `markdown`, `text`, and similar.
- Add it directly after opening backticks: use ` ```bash ` and not bare
 ` ``` `.
- If language is unclear, use `text` as fallback.

Examples:

Bad (missing language):

~~~markdown
```
ls -la
```
~~~

Good (explicit language):

~~~bash
ls -la
~~~

Reference: see `/.markdownlint.yaml` for current configuration.
